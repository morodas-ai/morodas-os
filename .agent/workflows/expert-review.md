# Expert Review Workflow（4段階方式）

MORODAS OS（`morodas-os-init`）に対する変更作業時は、以下の段階的プロセスを実行する。

---

## 使い方

どのチャットからでも以下のように呼び出せる:

```
/expert-review
```

呼び出すと、第1段階→第2段階→第3段階→第4段階の順に全ステージを実行し、結果を報告する。

### 部分実行

特定の段階だけ実行したい場合は指定できる:
- 「専門家の第1段階だけお願い」→ 基盤ガードのみ
- 「専門家の第3段階まで」→ 第1〜第3段階
- 「/expert-review」（指定なし）→ **全4段階をフル実行**

---

## 原則

- **「選ぶ」のではなく「検出する」**。各段階で全専門家の問いかけをスキャンし、該当するものを拾い上げる
- トリアージ（主観的選別）は禁止。自分の盲点を補う専門家ほど、自分が除外しがちだから
- 段階は「深さ」の違いであり、「範囲の絞り込み」ではない

## 第1段階：基盤ガード（5人）

## 第4段階：神の目（メタレビュー）

**発動条件**: 第1〜第3段階の完了後、または `/expert-review` でフル実行時
**所要時間**: 3〜5分
**目的**: 25人の専門家「全員が見落としている視点」を炙り出す。専門家の外側から全体を俯瞰する。

### 5つの問い

第1〜第3段階の全結果を踏まえた上で、以下の5問に回答する:

**問1：「全員が見ていない方向」は何か？**
> 25人の専門家は、それぞれの管轄内で問題を検出する。だが、どの専門家の管轄にも属さない「隙間」にこそ、致命的な問題が潜む。

**問2：「専門家同士の矛盾」はないか？**
> 異なる専門家の推奨が互いに矛盾していないかを検証する。矛盾がある場合、以下の優先順序で解決する:
> 1. **セキュリティ** > 2. **データ整合性** > 3. **ユーザー体験** > 4. **開発効率**
> 解決できない場合はオジキに判断を仰ぐ。

**問3：「1年後の自分」はこの判断をどう評価するか？**
> 今は正しく見える設計が、1年後にスケーリング・法人化・チーム拡大で破綻する可能性はないか？

**問4：「そもそもこの作業自体が不要」という可能性はないか？**
> この変更は本当にオジキの目標（月100万円／法人化／停滞防止）に貢献しているのか？

**問5：「この検証プロセス自体」は正しく機能しているか？**
> 専門家の問いかけは古くなっていないか？新しい視点を追加すべきか？段階の配置に見直しが必要か？

### 出力

```
### 🔮 第4段階：神の目

**問1 - 盲点**: [25人の誰もカバーしていない領域の指摘]
**問2 - 矛盾**: [専門家間の推奨の衝突と解決]
**問3 - 時間軸**: [1年後の視点からの評価]
**問4 - 不要性**: [この作業自体のROI判定]
**問5 - 自己点検**: [このプロセス自体の改善提案]
```

## チェック実績の記録

各フル実行後、以下を `.agent/experts/review-log.md` に追記する:

```markdown
## YYYY-MM-DD: [作業タイトル]
- **対象**: [変更したファイル/機能]
- **検出**: #X ○○の専門家 → [指摘内容]（対応: 修正済/保留/却下）
- **神の目**: [主な発見]
- **実績**: [実際に問題を防いだ専門家がいれば記録]
```

この実績ログにより:
- 一度も役に立っていない専門家 → 問いかけの見直し対象
- 頻繁に検出する専門家 → 第1段階への昇格候補
- 新パターンの発見 → 新しい専門家の追加候補

---

## 月次自己点検（毎月1日）

毎月1日に `review-log.md` を振り返り、以下を判定する:
1. 各専門家の検出実績（0回なら問いかけを見直し）
2. 新たに追加すべき専門家はいるか
3. 段階の配置は適切か
4. 専門家シリーズ原本（`.agent/experts/expert-series.md`）の更新
